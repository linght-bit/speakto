# Speak To — Архитектура проекта

## Что это за проект

Сейчас Speak To — это 2D-игра на canvas с голосовым управлением, статичной картой космического корабля и встроенным creative-режимом для ручной сборки карты.

Проект уже ушёл далеко от ранней структуры с домиком, колодцем и простыми квестами. Теперь основа проекта такая:

- один общий state игры;
- данные карты и игровых объектов лежат в JSON;
- логика разложена по системам;
- отрисовка вынесена в UI-слой;
- creative-режим работает как отдельный редактор поверх основной игры;
- pathfinding и проходимость завязаны на геометрию корпуса корабля и объекты карты.

## Главные принципы

### 1. Один state на всю игру

Источник правды — `core/gameState.js`.

Через него хранятся:

- позиция игрока;
- язык UI и игрока;
- инвентарь;
- флаги мира;
- карта объектов;
- состояние контейнеров и поверхностей;
- служебные флаги движения и интеракций.

Чтение:

```js
const state = getGameState();
const lang = getGameState('ui.language');
```

Изменение:

```js
updateGameState({
  player: { isMoving: true }
});
```

### 2. Тексты не живут в runtime-коде

Все пользовательские и служебные тексты должны лежать в `i18n/ru.json` и `i18n/pt.json`.

В коде допустимы только ключи:

```js
getText('messages.error');
getText('fox.not_understood');
```

Это правило касается:

- UI;
- сообщений F.O.X;
- ошибок загрузки;
- служебных console-логов bootstrap.

### 3. Данные мира отделены от логики

Файлы в `data/` отвечают на вопрос «что есть в мире».

Системы в `systems/` отвечают на вопрос «что с этим делать».

Примеры:

- `data/objects.json` хранит и метаданные мира, и статичную раскладку корабля;
- `systems/pathfindingSystem.js` решает, какие клетки проходимы;
- `systems/actionSystem.js` решает, как выполняются команды;
- `ui/render.js` решает, как это всё показать на экране.

### 4. Связь между частями игры идёт через события

Основной канал связи — `eventSystem`.

Примеры событий:

- `game:initialized`
- `game:state-changed`
- `voice:recognized`
- `action:executed`
- `action:failed`
- `door:opened`
- `item:taken`

Это нужно, чтобы системы не сцеплялись друг с другом слишком жёстко.

### 5. Карта корабля статичная по умолчанию

Базовая карта должна жить в `data/objects.json`.

Creative-режим нужен как рабочий редактор, но не как источник процедурной генерации.

Идея простая:

- рисуем руками;
- экспортируем;
- переносим в `data/objects.json`;
- игра стартует уже на готовой статичной карте.

## Как сейчас запускается игра

### Порядок загрузки

`index.html` подключает скрипты в таком порядке:

1. события и конфиг;
2. state;
3. i18n-утилиты;
4. игровые системы;
5. рендер и creative-режим;
6. bootstrap.

### Что делает bootstrap

`bootstrap.js` — это сборщик приложения в рантайме.

Он:

1. проверяет, что ключевые системы уже загружены;
2. читает JSON-файлы из `data/` и `i18n/`;
3. инициализирует i18n;
4. передаёт данные в системы;
5. грузит геометрию карты в pathfinding;
6. сбрасывает state к стартовому;
7. запускает голос;
8. запускает рендер;
9. подписывает bootstrap-логи на события.

Схема потока:

```text
index.html
   ↓
bootstrap.js
   ↓
data/*.json + i18n/*.json
   ↓
systems + state + renderer
   ↓
игра готова к работе
```

## Как устроена карта корабля

### Геометрия корпуса

`data/objects.json` в поле `map` описывает:

- ширину и высоту мира в клетках;
- левую и правую границу корпуса;
- носовую полукруглую часть;
- нижнюю границу корпуса.

Это нужно, чтобы:

- рисовать корабль;
- отличать пол, стену и открытый космос;
- правильно строить проходимость.

### Статичная карта

`data/objects.json` в поле `objects` хранит всё, что пользователь видит внутри корабля:

- внутренние стены;
- двери;
- крупные агрегаты;
- сундуки;
- окна;
- декоративные элементы.

### Проходимость

`systems/pathfindingSystem.js` собирает walkable-grid из трёх слоёв:

1. геометрия корпуса;
2. объекты карты;
3. runtime-исключения из creative-режима, например удалённые клетки внешней стены.

---

## Как устроен creative-режим

`ui/creativeMode.js` — это отдельный слой редактора, который работает поверх основной игры.

Он умеет:

- включаться только в dev-режиме;
- показывать палитру инструментов;
- ставить и удалять объекты;
- удалять клетки внешней стены;
- экспортировать и импортировать карту;
- рисовать сетку поверх мира;
- патчить рендер для custom-объектов редактора.

Важно:

- creative-режим не должен превращаться в отдельную систему мира;
- это инструмент сборки карты, а не второй источник данных;
- главный источник статичной карты всё равно `data/objects.json`.

## Что ещё стоит улучшить

### 1. Слишком много временных данных висит в `window`

Сейчас bootstrap складывает в `window` много отдельных кусков:

- `window.itemsData`
- `window.actionsData`
- `window.mapObjectsData`
- `window.ruTexts`
- `window.ptTexts`
- и так далее.

Это работает, но со временем становится непонятно:

- что реально используется;
- что осталось от старых этапов;
- что можно удалить без риска.

### Предлагаемое решение

Статус на текущий момент:

- Шаг 1 уже внедрён в `bootstrap.js`: появился единый контейнер `window.appContext`.
- Старые `window.*` поля пока оставлены как совместимость через переходный слой.
- Это позволяет мигрировать модули дальше без массовой ломки кода.

Без резкой ломки проекта следующий шаг должен быть таким:

#### Шаг 1. Ввести один реестр приложения (выполнено)

Вместо россыпи полей в `window` хранить данные в одном объекте, например:

```js
window.appContext = {
  data: {
    actions: ...,
    items: ...,
    mapObjects: ...,
    worldObjects: ...,
    map: ...,
    sprites: ...,
  },
  services: {
    eventSystem: ...,
    actionSystem: ...,
    voiceSystem: ...,
    pathfindingSystem: ...,
    gameRenderer: ...,
  }
};
```

#### Шаг 2. Оставить старые `window.*` только как временную совместимость (выполняется)

Сначала проект начинает читать данные из `appContext`, а старые глобальные поля живут как переходный слой.

#### Шаг 3. Постепенно убрать прямые обращения к лишним глобальным полям (следующий этап)

Не одной большой переделкой, а по частям:

- сначала `bootstrap.js`;
- потом `creativeMode.js`;
- потом `systems/*.js`.

Итог: архитектура останется простой, но станет заметно чище.

## Практическое правило для новых изменений

Если добавляется новая механика, нужно сразу ответить на 4 вопроса:

1. Где лежат её данные?
2. Где лежит её логика?
3. Где лежат её тексты?
4. Через что она общается с остальной игрой?

Правильный ответ обычно такой:

- данные: `data/`
- логика: `systems/` или `ui/`
- тексты: `i18n/`
- связь: `eventSystem` и `gameState`

## Коротко

Текущее ядро проекта выглядит так:

- `bootstrap.js` собирает приложение;
- `gameState.js` хранит правду о мире;
- `data/objects.json` хранит статичную карту корабля;
- `pathfindingSystem.js` отвечает за проходимость;
- `render.js` показывает мир;
- `creativeMode.js` помогает вручную строить карту;
- `i18n/*.json` хранят тексты;
- `rules.md` задаёт правила, по которым всё это развивается.

Это и есть текущая рабочая архитектура проекта.
