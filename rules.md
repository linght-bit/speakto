# SPEAK TO — RULES (ЧИСТАЯ АРХИТЕКТУРА)

Этот файл фиксирует обязательные правила разработки.
Цель: стабильная data-driven архитектура без legacy-отклонений.

---

## 1) Базовые инварианты

- Тексты не пишем в runtime-коде. UI/подсказки/ошибки только через i18n-ключи.
- Игровые данные не хардкодим в системах. Источник данных — только /data/*.json.
- Единый state хранится только в /core/gameState.js.
- Менять state можно только через updateGameState(...), без прямых мутаций объекта из getGameState().
- Модули связываются через /core/events.js, а не прямыми зависимостями по бизнес-логике.
- `rules.md` — read-only контракт, меняется только по прямому запросу владельца.

## 3) Языковая модель проекта

- Игрок даёт команды на португальском.
- Лисёнок отвечает на русском + фраза которую нужно сказать на бразильском.
- Управление персонажем — только голосовыми командами.
- все названия объектов на португальском
- все русские тексты хранятся только в ru.json
- все бразильские-португальские тексты хранятся только в pt.json
- в других файлах в коде нигде не используется русский и португальский СТРОГО

## 4) Правило распознавания намерения и сущности (обязательно)

- Этап 1: определить намерение (тип действия).
- Этап 2: определить сущность (конкретный предмет/цель/контейнер/поверхность/дверь).
- Если намерение распознано, а сущность конфликтует или не существует — действие не выполнять.
- Запрещён опасный fallback от уточнённой сущности к общей.
  Пример: "pega a chave cinza" нельзя превращать в "pega a chave".
- При отказе ассистент обязан дать подсказку:
  - что не найдено;
  - какие валидные варианты есть (если есть).

---

## 5) UI-инварианты (история голоса и лисёнка)

- История голосовых команд хранит до 100 записей.
- История подсказок лисёнка хранит до 50 записей.
- Оба окна поддерживают режимы Свернуть/Развернуть.
- В свернутом режиме показывается одна актуальная строка.
- В развернутом режиме показывается история.
- Выполненные голосовые команды визуально отличаются (успех/зелёный).

---

## 6) События и контракты

- События именуются в формате namespace:event (например, voice:commandExecuted).
- Event payload должен быть минимальным, но достаточным для подписчика.
- Новые события фиксировать рядом с местом эмита в комментарии или в PR-описании.
- Для отказов действия использовать структурированный failure-контекст (code + meta).

---

## 7) Логи и ошибки

- Отладочные console.log/console.warn в прод-коде не оставлять.
- Допускается console.error(error) для исключений.
- Пользовательские сообщения об ошибках — только через i18n и UI-события.

---

## 8) Проверка перед изменением

- Это текст? → i18n.
- Это данные игры? → /data.
- Это состояние? → gameState + updateGameState.
- Это логика правила/действия? → /systems.
- Это отображение? → /ui.

---

## 9) Что не раздувать в rules

- Не дублировать большие код-примеры "правильно/неправильно" для каждого пункта.
- Не перечислять устаревшие или отсутствующие файлы.
- Не переносить сюда временные задачи и баг-репорты.
- Детали реализации хранить в коде и тестах, в rules оставлять только инварианты.

КАК РЕАЛИЗУЮТСЯ КВЕСТЫ , СВОБОДА ДЕЙСТВИЙ И ВОЗМОЖНОСТЬ ПЕРЕСКАКИВАТЬ ВПЕРЕД ПО СЮЖЕТУ 
есть несколько слоёв:

важные квестовые предметы и мы не можем их перескочить по квесту. например книга которую нас просят принести.
(это state-гейты — проверка наличия состояния/предмета)
важные квестовые действия, например открыть дверь в новое помещение.
(это action-гейты — проверка совершённого действия)

мы не можем перескочить эти важные этапы.
Если игрок делает действие которое по квесту запланировано позже этих этапов, то подсказка должна вести на момент до начала логики этого этапа.
(подсказка всегда ведёт к конкретному правильному действию — возвращает на нужный шаг)

важное дополнение по логике:

перед выполнением любого действия всегда проверяются гейты
если есть незакрытый обязательный гейт → действие не выполняется → показывается подсказка
один гейт = одна причина отказа (не показываем несколько сразу)
ключевой момент 2 квеста - наличие костюма: что бы взять костюм надо открыть сундук - это момент ключевой для квеста - подсказка должна нас направить туда и тут каждый раз индивидуально прописывается дополнительный диалог для ситуации когда игрок "перескакивает"
В квесте 2 при перескоке (если у игрока нет костюма - но он уже начинает действия которые стоят в квесте после получения костюма) мы говорим ему:
Тебе бы для начала переодеться - посмотри нет ли вещей в ящике: abre a baú.
ключевой момент для 2 квеста, если костюм есть - наличие ключа.
Если костюм есть а игрок перескакивает и делает действия которые по квесту стоят после получения ключа - мы пишем ему:
Проверь карманы, возможно в них есть ключ? olhar os bolsos

так для каждого квеста будем определять ключевые точки.
(гейты должны быть упорядочены по логике прохождения — сверху вниз)

а в остальном должна быть свобода и правила:
Все действия доступны всегда если условия выполнены.
