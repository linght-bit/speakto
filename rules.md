# ГЛАВНЫЕ ПРАВИЛА АРХИТЕКТУРЫ

Этот файл описывает обязательные правила разработки для проекта Speak To.

---

## 1. Запрещено писать тексты в коде

Нельзя использовать строки в logic, UI и systems файлах. Все тексты только через i18n.

```javascript
// ❌ НЕПРАВИЛЬНО
console.log('Привет, мир!');
foxSay('Что-то случилось');

// ✅ ПРАВИЛЬНО
console.log(getText('ui.greeting'));
foxSay(getText('fox.warning'));
```

---

## 2. Все тексты только через i18n

Использовать только ключи. Тексты хранятся только в `/i18n/ru.json` и `/i18n/pt.json`.

```javascript
getText('ui.welcome', 'ru')        // Русский текст
getText('dialogue.hello', 'pt')    // Португальский текст
getText('quests.title')            // Текущий язык
```

---

## 3. Логика не содержит данные

Нельзя хардкодить квесты, NPC, предметы или локации. Все данные берутся из `/data`.

```javascript
// ❌ НЕПРАВИЛЬНО
const quests = [
  { id: 'quest1', title: 'Первый квест' },
  { id: 'quest2', title: 'Второй квест' }
];

// ✅ ПРАВИЛЬНО
const quests = await fetch('./data/quests.json').then(r => r.json());
```

---

## 4. Один источник состояния

Весь state находится только в `/core/gameState.js`.

```javascript
// ❌ НЕПРАВИЛЬНО
let playerInventory = [];
let playerHealth = 100;
let isGameRunning = true;

// ✅ ПРАВИЛЬНО
const state = getGameState();
state.player.inventory         // Инвентарь
state.player.health            // Здоровье
state.game.running             // Статус игры
```

---

## 5. Изменение состояния только через gameState.js

Нельзя изменять state напрямую. Нельзя создавать новые переменные состояния вне gameState.

```javascript
// ❌ НЕПРАВИЛЬНО
getGameState().player.health = 50;
const globalState = {};

// ✅ ПРАВИЛЬНО
updateGameState({
  player: { health: 50 }
});
```

## АРХИТЕКТУРА

### STATE — /core/gameState.js

Содержит только:
- прогресс игрока
- флаги
- активные квесты
- текущая сцена

**Запрещено:**
- логика
- тексты
- описание квестов

### DATA — /data/*.json

Файлы:
- `quests.json` — описание квестов
- `locations.json` — структура локаций
- `items.json` — параметры предметов

Содержит:
- описание квестов
- структуру локаций
- параметры предметов

**Только данные, без логики.**

### I18N — /i18n/*.json

Файлы:
- `ru.json` — русские тексты
- `pt.json` — португальские тексты

Содержит только тексты. Никакой логики.

### SYSTEMS — /systems/*.js

Файлы:
- `questSystem.js` — логика квестов
- `dialogueSystem.js` — логика диалогов
- `inventorySystem.js` — логика инвентаря

Содержит:
- правила
- обработку событий
- игровую логику

**Запрещено:**
- тексты
- хардкод данных

### UI — /ui/*.js

Файлы:
- `render.js` — отрисовка и интерфейс

Содержит:
- отрисовку
- интерфейс
- экраны

Использует только ключи i18n.

### UTILS — /utils/*.js

Файлы:
- `getText.js` — получение текстов
- `helpers.js` — вспомогательные функции

Содержит вспомогательные функции.

### CONFIG — /config/gameConfig.js

Содержит:
- стартовые параметры
- баланс
- глобальные настройки

### EVENTS — /core/events.js

Система событий для связи между модулями.

## ДОПОЛНИТЕЛЬНЫЕ ПРАВИЛА

- Лисёнок говорит только по-русски.
- Игрок говорит только по-португальски.
- Управление только голосом.
- Архитектура строится сразу правильно, без последующих переделок.

### Правило распознавания намерения и сущности (обязательно)

- Распознавание команды всегда разделяется на 2 независимых этапа:
  - этап 1: определить **намерение** (глагол/тип действия);
  - этап 2: определить **сущность** (конкретный предмет/объект/цель).
- Если намерение распознано, но сущность распознана с конфликтом или не существует в данных,
  выполнение действия запрещается.
- Запрещён «опасный fallback» от специфичной сущности к более общей,
  если в команде есть уточняющий признак.
  Пример: "pega a chave cinza" нельзя автоматически превращать в "pega a chave".
- При отказе система обязана вернуть подсказку через лисёнка:
  - что именно не найдено/не совпало;
  - какие валидные варианты доступны сейчас (если есть).
- Это правило универсально и применяется ко всем типам сущностей:
  предметы, контейнеры, двери, цели подхода, поверхности.

## ПРИНЦИПЫ

- Data-driven архитектура.
- Лёгкое добавление новых квестов, NPC, предметов, локаций и языков через данные.
- Запрещён хардкод.
- Запрещено дублирование.

---

## ПРОВЕРКА ПЕРЕД ДЕЙСТВИЕМ

Если это **текст** → использовать i18n  
Если это **данные** → размещать в /data  
Если это **состояние** → использовать gameState  
Если это **логика** → размещать в systems  
Если это **отображение** → размещать в UI  

**Если есть сомнение — следовать архитектуре.**
